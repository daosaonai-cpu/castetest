1. Introduction & Overview

1.1 What is an Operating System?

An Operating System (OS) is the most critical system software running on a computer. It acts as an intermediary between the user of a computer and the computer hardware. Without an OS, every program would need to include its own code to manage hardware components like the CPU, memory, and disk drives.

The Dual Role of an OS

  1. Resource Manager: The OS acts as a fair referee, allocating resources (CPU time, memory space, file storage, I/O devices) to specific programs and users as necessary for their tasks. It handles conflicts and ensures efficiency.
  2. Control Program: It controls the execution of user programs to prevent errors and improper use of the computer. It provides a consistent, abstracted interface for applications to interact with hardware.

1.2 A Brief History of Operating Systems

  • First Generation (1945-1955): Vacuum Tubes and Plugboards. No OS existed. Programs were written in machine language and manually wired into the computer.
  • Second Generation (1955-1965): Transistors and Batch Systems. Introduction of batch processing. Jobs with similar needs were batched together to speed up processing. The OS was simple, mostly just a transfer program.
  • Third Generation (1965-1980): ICs and Multiprogramming. Introduction of Multiprogramming (keeping multiple jobs in memory to keep the CPU busy) and Spooling (Simultaneous Peripheral Operations On-Line). Time-sharing systems emerged, allowing interactive computing. UNIX was developed.
  • Fourth Generation (1980-Present): Personal Computers. GUI (Graphical User Interface) became standard. Network operating systems and distributed operating systems developed. Examples: Windows, macOS, Linux, Android, iOS.

1.3 OS Booting Process (Example)

When you press the power button, the computer goes through a specific sequence:

1. Power On Self Test (POST) by BIOS/UEFI.
2. BIOS/UEFI locates the Boot Device (e.g., Hard Drive, USB).
3. BIOS/UEFI loads the Master Boot Record (MBR) or EFI System Partition.
4. The Bootloader (e.g., GRUB, Windows Boot Manager) is executed.
5. The Bootloader loads the OS Kernel into Main Memory (RAM).
6. The Kernel initializes hardware, starts background services (daemons), and mounts the root file system.
7. User Space initialization (e.g., presenting the login screen or desktop).

2. Process Management & Scheduling

A program by itself is not a process; a program is a passive entity (a file containing instructions), whereas a process is an active entity, with a program counter specifying the next instruction to execute.

2.1 The Process Control Block (PCB)

The OS maintains information about every process in a data structure called the PCB. It contains:

  • Process State (e.g., new, ready, running)
  • Program Counter (address of next instruction)
  • CPU Registers (contents of registers when interrupted)
  • CPU Scheduling Information (priority)
  • Memory Management Information (page tables)
  • Accounting Information (CPU used, time limits)
  • I/O Status Information (devices allocated)

2.2 Process States and Transitions

New Ready Running Terminated

Processes can also move from 'Running' to 'Waiting/Blocked' if they need I/O, and then back to 'Ready'.

2.3 CPU Scheduling Algorithms Explained

The CPU scheduler decides which process in the Ready queue gets the CPU next. The goal is to optimize CPU utilization, throughput, turnaround time, waiting time, and response time.

First-Come, First-Served (FCFS)

Simplest algorithm. Non-preemptive. Processes are served in the exact order they arrive. Can suffer from the Convoy Effect: short processes wait for one long process to finish.

Shortest Job First (SJF)

Selects the process with the smallest next CPU burst. Provides minimum average waiting time. Can be preemptive (Shortest Remaining Time First) or non-preemptive. Risk: Starvation of long processes.

Round Robin (RR)

Designed for time-sharing systems. Each process gets a small unit of CPU time (time quantum, e.g., 10-100ms). Preemptive. If a process doesn't finish in its quantum, it's moved to the back of the Ready queue.

Multilevel Queue Scheduling

Partitions the ready queue into multiple separate queues (e.g., foreground interactive processes vs. background batch processes). Each queue has its own scheduling algorithm. Queues are prioritized.

3. Concurrency & Synchronization

When multiple processes execute concurrently, they may need to share data or hardware. If access is not controlled, data inconsistency occurs.

3.1 The Critical Section Problem

A Critical Section is a segment of code where a process accesses shared resources (variables, files, etc.). The problem is ensuring that when one process is in its Critical Section, no other process is allowed into its Critical Section.

A solution to the Critical Section problem must satisfy three requirements:

  1. Mutual Exclusion: Only one process can be in the critical section at a time.
  2. Progress: If no process is in the critical section, processes waiting to enter should get a chance.
  3. Bounded Waiting: There must be a limit on how many times other processes can enter their critical sections after a process has made a request to enter its own.

3.2 Semaphores and Mutexes

Hardware solutions are complex. OS provides software tools:

  • Mutex Locks (Mutual Exclusion): A simple lock. A process must acquire the lock before entering the critical section and release it when exiting. Usually implemented with hardware instructions like 'test_and_set'.
  • Semaphores: An integer variable accessed via two atomic operations: wait() (decrements) and signal() (increments).
    • Binary Semaphore: Acts like a mutex (values 0 or 1).
    • Counting Semaphore: Controls access to a resource with multiple instances.

3.3 Deadlocks

A deadlock is a situation where a set of processes are blocked because each process is holding a resource and waiting for another resource acquired by some other process.

4 Necessary Conditions for Deadlock:

  • Mutual Exclusion: At least one resource must be non-sharable.
  • Hold and Wait: A process holding a resource is waiting to acquire additional resources held by others.
  • No Preemption: Resources cannot be forcibly taken away from a process.
  • Circular Wait: A closed chain of processes exists, each waiting for a resource held by the next.

Handling Deadlocks:

  • Prevention: Ensure at least one of the 4 conditions cannot hold.
  • Avoidance: OS requires advance knowledge of resource requests (e.g., Banker's Algorithm) to ensure the system never enters an "unsafe" state.
  • Detection & Recovery: Allow deadlocks to happen, detect them periodically, and recover (e.g., by aborting processes).
  • Ignorance: (The Ostrich Algorithm) Pretend they never happen. Common in modern desktop OSes like Windows/Linux because deadlocks are rare and prevention is too costly.

4. Memory Management & Virtual Memory

Memory management manages primary memory (RAM) and moves processes back and forth between main memory and disk during execution. It keeps track of every memory location.

4.1 Contiguous vs. Non-Contiguous Allocation

  • Contiguous: The entire process is loaded into a single block of memory. Suffers from External Fragmentation (plenty of free memory total, but not in a single contiguous block).
  • Non-Contiguous (Paging & Segmentation): A process can be scattered across physical memory. Solves external fragmentation.

Paging Explained

Physical memory is divided into fixed-size blocks called Frames. Logical (virtual) memory is divided into blocks of the same size called Pages.

When a process runs, its pages are loaded into any available memory frames. The OS maintains a Page Table for every process to map logical page numbers to physical frame numbers.

  • Solves external fragmentation.
  • Introduces Internal Fragmentation (the last page of a process might not be fully used).

4.2 Virtual Memory

Virtual memory is a technique that allows the execution of processes that are not completely in memory. It provides the illusion of a very large memory, often much larger than actual physical RAM.

  • Demand Paging: Pages are brought into memory only when they are needed (demanded) by the CPU.
  • Page Fault: If the CPU tries to access a page that is NOT currently in RAM (it's on the disk), a page fault trap occurs. The OS must suspend the process, read the page from disk into a free frame, update the page table, and restart the instruction.
  • Page Replacement Algorithms: If memory is full, the OS must swap out an existing page to make room for a new one. Goal: Minimize page faults.
    • FIFO (First-In, First-Out): Replaces the oldest page. Can suffer from Belady's Anomaly (more frames = more faults).
    • LRU (Least Recently Used): Replaces the page that has not been used for the longest time. Generally very good performance.
    • Optimal: Replaces the page that will not be used for the longest period of time in the future. (Theoretical limit, used as a benchmark).
// Thrashing If a system does not have enough memory, it might spend more time swapping pages in and out of the disk than actually executing processes. CPU utilization drops drastically. This severe performance collapse is called Thrashing.

5. File System Implementation

The file system provides a logical view of information storage and is implemented on secondary storage (disks). It organizes data into files and directories for easy retrieval.

5.1 Directory Structure

Directories organize files. Common structures include:

  • Single-Level: All files in one directory (naming collisions).
  • Two-Level: Separate directory for each user.
  • Tree-Structured: Hierarchical structure (like Windows C:\ or Unix /). Most common today. Allows arbitrary depth.
  • Acyclic-Graph: Allows directories to share subdirectories and files (links/shortcuts).

5.2 Allocation Methods

How do we allocate disk space to a file so it can be accessed efficiently?

Contiguous

File occupies contiguous blocks on disk. Great for sequential access. Bad for file growth; causes external fragmentation.

Linked

File is a linked list of blocks. Scattered anywhere. Solves fragmentation, file growth is easy. Bad for random access (must traverse list).

Indexed

Brings all pointers together into an 'index block'. Supports direct access. Slight overhead for the index block itself.

5.3 Free-Space Management

The OS must track which disk blocks are free. Common methods:

  • Bit Vector (Bitmap): 1 bit per disk block (1 = free, 0 = allocated). Fast to find contiguous free blocks, but bitmap can be large.
  • Linked List: Link together all free blocks. No extra space needed, but traversing is slow.

Unix Inodes:

In Unix-like systems, an inode (index node) is a data structure representing a file or directory. It stores metadata (permissions, ownership, size, timestamps) and pointers to the actual data blocks on the disk using a combined indexed scheme (direct pointers, single-indirect, double-indirect, triple-indirect) to handle both small and huge files efficiently.
